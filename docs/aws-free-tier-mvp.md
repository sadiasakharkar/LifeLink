# LifeLink Production MVP Notes

## DynamoDB Schema Design

### `LifeLinkUsers`
- Primary key: `id` (String)
- Attributes:
  - `id`
  - `name`
  - `email`
  - `passwordHash`
  - `role`
  - `hospitalId`
  - `hospitalName`
  - `createdAt`
  - `updatedAt`
- Recommended GSI for production scale:
  - `email-index` on `email`

### `LifeLinkDonors`
- Primary key: `id` (String)
- Attributes:
  - `id`
  - `bloodGroup`
  - `organType`
  - `status`
  - `hospitalId`
  - `hospitalName`
  - `location`
  - `createdAt`
  - `updatedAt`

### `LifeLinkRecipients`
- Primary key: `id` (String)
- Attributes:
  - `id`
  - `name`
  - `bloodGroup`
  - `organType`
  - `urgency`
  - `waitingTime`
  - `status`
  - `hospitalId`
  - `hospitalName`
  - `createdAt`
  - `updatedAt`

### `LifeLinkAllocations`
- Primary key: `id` (String)
- Attributes:
  - `id`
  - `donorId`
  - `recipientId`
  - `status`
  - `score`
  - `explanation`
  - `rankingList`
  - `approvedBy`
  - `approvedAt`
  - `rejectedAt`
  - `approvalNotes`
  - `donorSnapshot`
  - `recipientSnapshot`
  - `timestamps`
  - `createdAt`
  - `updatedAt`

### `LifeLinkAuditLogs`
- Primary key: `id` (String)
- Attributes:
  - `id`
  - `userId`
  - `action`
  - `entityType`
  - `entityId`
  - `timestamp`
  - `metadata`

## Approval Workflow

1. Donor is created.
2. Matching service creates a `PENDING` allocation.
3. Admin or Doctor reviews it in the Approval Panel.
4. Allocation is moved to:
   - `APPROVED`
   - `REJECTED`
5. Audit log entry is automatically recorded.

## Static / Mocked But Functional UI Features

- Allocation ranking list is mock-scored but generated by the backend.
- Notifications show realistic toasts in the UI.
- Transport tracking uses static operational updates from the backend response.

## AWS Free Tier Deployment Steps

### Backend
1. Create the five DynamoDB tables in on-demand or low-throughput mode.
2. Package the Express backend as Lambda using `backend/src/handler.js`.
3. Create an HTTP API in API Gateway and point the default route to the Lambda.
4. Set Lambda environment variables:
   - `JWT_SECRET`
   - `AWS_REGION`
   - `USERS_TABLE`
   - `DONORS_TABLE`
   - `RECIPIENTS_TABLE`
   - `ALLOCATIONS_TABLE`
   - `AUDIT_LOGS_TABLE`
   - `CLIENT_URL`
5. Attach an IAM policy giving Lambda access only to those DynamoDB tables.

### Frontend
1. Build the frontend with `npm run build`.
2. Create an S3 bucket for static website hosting.
3. Upload `frontend/dist` contents to the bucket.
4. Set `VITE_API_URL` to the API Gateway URL before build.
5. Optionally place CloudFront in front of S3 for HTTPS and caching.

### Local / Dev Notes
- Use `DYNAMODB_ENDPOINT` to point the backend at DynamoDB Local during development.
- Keep `SEED_DEMO_DATA=true` for a demo-friendly first run.
